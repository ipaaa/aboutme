## Issues — Workflow

- **Eleventy double-emits `.html` passthroughCopy targets.** `src/googleb524dd295ffe1b52.html` got rendered as a Nunjucks template *and* passthroughCopied, producing both `_site/googleb524dd295ffe1b52.html` and `_site/googleb524dd295ffe1b52/index.html`. Both byte-identical, GSC verification works fine, but tidy-up via `eleventyConfig.ignores.add("src/googleb524dd295ffe1b52.html")` is available if it bugs the captain later.
- **GSC verification UX gap.** Captain accidentally pasted the HTML filename (`googleb524dd295ffe1b52.html`) into `seo.json`'s `googleVerificationToken` field instead of an actual meta-tag token. The conditional in `base.njk` would have happily emitted a broken `<meta name="google-site-verification" content="googleb524dd295ffe1b52.html">` if deployed. Mitigation: `seo.json`'s `_comment_googleVerificationToken` field is explicit about which method it's for. Could be sharpened with a startup-time validator (e.g. `base.njk` only emits if token doesn't end with `.html`) but optional.

## Observations

- **Bilingual identity design has a duplication trap.** T-1, T-2, T-3, and T-4 tagline candidates all overlapped with About Me's role-list (Writer / Documentary / g0v co-founder). Captain caught this at preview; the eventual place-anchor tagline ("Between the SF Bay Area and Taipei.") sidestepped it cleanly. Worth remembering: hero copy and About Me copy should *announce different things*, not say the same thing twice.
- **`prefers-color-scheme` is "correct behavior" but a UX trap for the site owner.** Auto-following the visitor's OS preference made captain's own site look unrecognizable on their own machine. If dark mode ever returns, opt-in toggle (not auto-follow) is the right pattern.
- **Tagline iteration cost was higher than the value.** Four cycles to land on "Between the SF Bay Area and Taipei." (T-3 bilingual → English place → `SF Bay` → `SF Bay Area` → `the SF Bay Area`). Three of those four were typo/grammar fixes that didn't need a full ensign dispatch cycle. A "captain direct edit" path for small string changes inside an active worktree would save round-trips.
- **GSC verification UX defaults push toward HTML-file upload.** Our `seo.json`/meta-tag scaffolding (#27) was built for the HTML-tag method, but GSC's default first-option happened to be HTML-file. Worth supporting both shapes — either via clearer template-comment guidance or by auto-detecting `.html`-suffix tokens and switching emission mode.
- **Build pipeline investment paid for itself in the same session.** #24 (Eleventy) shipped, then #22 (JSON-LD) and #27 (sitemap/robots/verification meta) immediately leveraged the new `<head>` template + passthroughCopy. The "is this build infra worth it for one page?" question became "yes" once two follow-ons landed within the hour.
- **FO/ensign separation worked best for design moments (3 hero directions, 3 visual-polish directions, 4 build-pipeline directions) — added friction for typo-grade edits.** The cycle pattern for tiny string changes was "FO writes brief → SendMessage to alive implementer → implementer edits 1 string → done signal → FO advances → captain confirms." For trivial edits this is heavier than warranted; could be shortened with FO-direct-edit-on-worktree for changes under N bytes.
- **Schema.org validation arrived clean.** ProfilePage + Person detected by Google Rich Results Test on first try, zero errors / warnings. Confirmed the ideation choice (W-3 over W-1/W-2) was right for personal/author pages.
